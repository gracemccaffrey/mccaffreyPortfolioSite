HealthHub

A health literacy and awareness platform that helps underserved communities find care and understand healthcare basics.

RoleFront-End / Full-Stack Developer
TimelineJan – Apr 2025
Team4 Developers (13-person cohort)
PlatformProgressive Web App

Problem Statement

Many individuals face barriers in understanding healthcare options, navigating insurance, and locating nearby providers. These challenges are especially prominent in underserved areas, where provider shortages and unreliable internet access can delay treatment and worsen outcomes.

Limited provider access

Long travel distances and shortages of care options.

Low health literacy

Medical terms and care processes can be confusing.

Insurance & cost barriers

Uncertainty around coverage and affordability.

Connectivity constraints

Limited broadband reduces access to online resources.

Research & Context

Social & Cultural Context

Research into rural Ohio communities shaped several core design decisions. Lower trust in external systems and limited technology adoption among some populations -- including communities where smartphone use is minimal -- meant the platform needed to feel familiar and straightforward rather than novel or complex. We prioritized plain-language content, predictable navigation, and a design aesthetic that aligned with established healthcare website conventions to build immediate credibility with users who might otherwise disengage.

Physical Constraints

Many users in the target demographic rely on cellular data rather than broadband, and some areas of rural Ohio have no reliable connectivity at all. This directly drove the decision to build a PWA with offline fallback functionality -- not as a feature addition, but as a core requirement. Pages were kept lightweight, images minimal, and the provider dataset was scoped to central Ohio specifically to keep the offline JSON file small enough to load without performance impact on older devices.

Health Literacy Gaps

A significant portion of the target population struggles to navigate healthcare systems, understand medical terminology, or know what to expect from an appointment. Rather than presenting raw provider data, we built dedicated Terminology and FAQ sections with plain-language definitions and answers to common questions about insurance, paperwork, and appointments -- all delivered through a dynamic content injection system that kept the interface clean and scannable.

Our Solution

Interactive Provider Map

Search providers by specialty and ZIP code, explore results on a map, and view provider details in expandable cards.

Health Literacy Tools

Terminology and FAQs help users understand common healthcare concepts, insurance basics, and what to expect at appointments.

Offline Fallback Dataset

When internet is unavailable, users can still search a curated central Ohio provider dataset by ZIP code -- a deliberate scoping decision that balances offline accessibility with app performance.

Mobile-First, Accessible UI

Designed for small screens and low-bandwidth environments with clear hierarchy and readable typography.

Modular Health Content System

FAQ and terminology content is stored in external JSON files and injected dynamically, keeping content updates completely separate from the HTML structure.

Provider Data Pipeline

Provider results are filtered, deduplicated, and scoped to keep search results clean, relevant, and fast -- even when pulling from a large government dataset.

Key Design Decisions

Platform Choice: PWA Website

Chose a website-style progressive web app for broad accessibility (no installation) and easier maintenance.

Maps: Google Maps API

Selected for geospatial accuracy, reliable routing/navigation support, and smooth mobile integration.

Offline Strategy: Curated Dataset

A full-state dataset was too large to load without compromising performance on older, low-bandwidth devices. We scoped the offline fallback to central Ohio -- the primary target demographic -- balancing accessibility with app speed.

Why it matters

Every decision on this project was made with a specific user in mind -- someone in a rural area, on an older phone, with inconsistent connectivity, who may not be comfortable navigating complex interfaces. The platform choices reflect that reality at every layer, from data architecture down to typography.

Technical Architecture

Client (Browser / PWA) → UI + Filtering
NPPES API (CMS) → Provider Data
Google Maps API → Geolocation + Map
Local Fallback JSON → Offline Search

The client handles all filtering, deduplication, and content injection directly in the browser -- there is no custom backend. This was a deliberate choice to keep the platform free to host, easy to maintain, and functional without a server dependency. Provider data flows from the NPPES API at search time, with the local JSON fallback stepping in automatically when connectivity is unavailable.

Tech Stack

  • Frontend: HTML, CSS, JavaScript
  • Data: NPPES/CMS API + curated JSON fallback
  • Mapping: Google Maps API
  • Architecture: Client-side filtering pipeline, provider deduplication logic, dynamic content injection, offline-first fallback strategy

Engineering Highlights

The platform was originally designed around the Google Places API for provider discovery. During development it became clear that Places lacked the structured data HealthHub needed -- no taxonomy codes, specialty classifications, or facility type metadata. We pivoted to the NPPES API provided by CMS, which offered all of this in a structured format. The tradeoff was that NPPES returns far more noise than Places, which is what drove the need for the filtering pipeline built on top of it.

The NPPES API returns many entries outside the scope of providers users would realistically visit. We filtered results using a curated taxonomy code whitelist covering primary care physicians, OB/GYNs, pediatricians, mental health professionals, hospitals, and urgent care clinics -- eliminating administrative offices, suppliers, and specialty facilities outside the platform's scope.

To reduce clutter, we merged entries representing the same organization by normalizing names and grouping providers within a small geographic radius.

Results were limited to providers within a 30-mile radius and capped at 20 provider groups per search. This kept the interface fast and navigable in low-bandwidth environments while preventing information overload for users who may not be comfortable parsing large result sets.

FAQ and terminology content is stored in structured external JSON files and injected dynamically via JavaScript, so content updates never require touching the HTML. This also ensures every entry follows a uniform expandable structure, improving consistency and accessibility across both pages.

Reflection & Future Work

Reflection

HealthHub taught me what it actually feels like to solve a problem, feel completely relieved, and then turn around and find yourself standing in front of a bigger one.

When the Google Places API fell short I didn't want to go to the team empty handed. I did the research on my own first, diving into healthcare data sources that none of us had any experience with, and found the NPPES/CMS API before bringing it to the group with an actual recommendation. Once we made the switch though, the relief didn't last long. I integrated the new API, ran it for the first time, and was immediately hit with an overwhelming number of results. Suppliers, administrative offices, duplicate entries, specialists that had nothing to do with what we were building. I had solved one problem and walked right into a harder one. The filtering pipeline and deduplication logic I built after that came directly from that moment of realizing the data just couldn't be trusted at face value.

Taking ownership of the UX and frontend pushed me in ways I didn't totally expect going in. Healthcare sites have to earn trust before a user reads a single word, and that trust lives in the design. I was designing for a community that needed something that felt real and approachable, that worked on older phones, that didn't assume any technical comfort level. There were so many things to hold at once and I loved every second of it. That is exactly the kind of challenge I am here for, the kind that asks me to be creative and analytical at the same time.

If I could go back and change things there are two I keep coming back to. The first is the visual direction. We used orange and went for a younger, more modern look to set ourselves apart a little, and while that landed well with the younger people we spoke to during research, I think it worked against us with older users who are honestly the ones who need this resource the most. Trust is easy to lose through a design decision and I think about that one a lot. The second is the data architecture. Building toward a database backed system from the start would have made everything so much more adaptable down the line. Region specific FAQs, multilingual support, localized content, none of that scales well out of static JSON files. It was the right call for the scope and timeline we had but if I was starting over I would build the foundation with more of that long term thinking in mind.

What made this feel like more than just a class project was that our end user was never just a concept. We talked to real people living in rural central Ohio. We understood what their days actually looked like and what was getting in the way of them accessing care. That changed how I approached every single decision. It stopped being about what would impress a professor and started being about what would actually work for someone who needed it. I care a lot about using what I can do to help people and this project is the clearest example I have of that. I am really proud of how real it looks, how far we went beyond what was required, and honestly what it showed me I was capable of.

Looking Ahead

HealthHub is functional and I am proud of what it is right now, but there is a clear path to what it could become.

The most immediate need is expanding the offline dataset beyond central Ohio. The current fallback c overs 40 providers across one region -- meaningful for the community we built it for, but a significant limitation for anyone outside that area. Broadening that coverage would be the first step toward making the platform a genuinely national resource.

The data architecture change I mentioned in my reflection would unlock a lot of what comes next. Moving from static JSON files to a database backed system would make it possible to serve region specific content -- FAQs and terminology tailored to the barriers that actually exist in a given area, because a rural community in Appalachian Ohio faces different challenges than one in rural New Mexico.

Multilingual support is another natural next step. The platform currently only serves English speakers, which immediately excludes a meaningful portion of the rural communities it was built for.

On the features side, more granular specialty filtering, appointment navigation guidance, and patient reviews would all make the provider search more useful. And a partnership with a public health organization or community health system would solve the hardest long term problem -- keeping the provider data accurate without manual updates.

The foundation is there. These are not rebuilds, they are extensions of decisions that were already made with this kind of growth in mind.
